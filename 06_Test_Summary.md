# API Test Summary

## 1. Общая информация

Проект: API тестирование Restful-Booker  
Тип проверки: ручное API-тестирование  
Инструмент: Postman  
Формат проверки: позитивные и негативные сценарии, проверка авторизации, CRUD-операции, анализ response behavior

---

## 2. Цель проверки

Целью проверки было:

- подтвердить базовую работоспособность API;
- проверить корректность CRUD-операций для booking;
- проверить работу авторизации;
- убедиться, что защищённые операции не выполняются без token;
- оценить поведение API при невалидных данных и несуществующих ресурсах.

---

## 3. Что было проверено

### 3.1 Авторизация

- получение token с валидными credentials;
- попытка получения token с невалидным password.

### 3.2 Чтение данных

- получение списка booking id;
- получение booking по существующему id;
- получение booking по несуществующему id.

### 3.3 Создание booking

- создание booking с валидным request body;
- проверка структуры ответа;
- сохранение booking_id для дальнейших операций.

### 3.4 Полное обновление booking

- PUT с валидным token;
- PUT без token;
- PUT для несуществующего booking_id.

### 3.5 Частичное обновление booking

- PATCH с валидным token;
- PATCH без token;
- PATCH для несуществующего booking_id.

### 3.6 Удаление booking

- DELETE с валидным token;
- DELETE без token;
- DELETE для несуществующего booking_id;
- дополнительная проверка результата удаления через повторный GET.

---

## 4. Подготовленные артефакты

В рамках проекта были подготовлены:

- описание API;
- scope и карта рисков;
- API checklist;
- API test cases;
- observations по негативным сценариям;
- Postman collection;
- notes по QA-сессии;
- итоговый API test summary.

---

## 5. Результаты проверки

По итогам выполненной проверки:

- базовые CRUD-операции API отрабатывают корректно;
- token выдаётся при валидной авторизации;
- невалидная авторизация не приводит к получению валидного token;
- защищённые операции не выполняются без token;
- запросы к несуществующим ресурсам обрабатываются предсказуемо;
- create / update / patch / delete работают корректно в рамках проверенного scope.

---

## 6. Найденные дефекты

В рамках текущего этапа проверки подтверждённых дефектов не найдено.

Это означает, что:

- API в рамках проверенных сценариев ведёт себя корректно;
- негативные сценарии обрабатываются предсказуемо;
- явных нарушений авторизации, целостности CRUD-логики или обработки отсутствующих ресурсов не выявлено.

---

## 7. Наблюдения

В рамках проекта были зафиксированы API observations по негативным сценариям:

- невалидная авторизация;
- запрос несуществующего booking_id;
- update / patch / delete без token;
- update / patch / delete для несуществующего booking_id.

По результатам проверки эти сценарии не были классифицированы как дефекты, так как API продемонстрировал ожидаемое и логичное поведение.

---

## 8. Что пока не проверено глубоко

На текущем этапе не были глубоко проверены:

- граничные значения всех полей booking;
- полные матрицы невалидных request body;
- нефункциональные характеристики API;
- нагрузка;
- безопасность в глубину;
- автоматизация API-проверок;
- локальный запуск платформы;
- расширенные интеграционные сценарии.

---

## 9. Основные выводы по качеству API

На основании проведённой проверки можно сделать следующие выводы:

### 9.1 Авторизация

Авторизация работает предсказуемо и не допускает невалидный доступ в рамках проверенных сценариев.

### 9.2 CRUD-поведение

Создание, чтение, обновление, частичное обновление и удаление booking отрабатывают корректно в рамках проверенных кейсов.

### 9.3 Обработка ошибок

API ожидаемо реагирует на невалидные условия: отсутствие token, неверные credentials, несуществующие ресурсы.

### 9.4 Общая оценка

Проверенный API-контур выглядит стабильным и логичным в рамках текущего scope.

Этот проект показал практические навыки ручного API-тестирования:

- работа с Postman;
- проверка авторизации;
- тестирование CRUD-операций;
- негативные проверки;
- анализ response behavior;
- умение отделять defect от корректного защитного поведения API.

Несмотря на отсутствие подтверждённых дефектов, проект является полноценным QA-кейсом, так как демонстрирует системную проверку API, корректную классификацию результатов и зрелый подход к качеству.
