# Scope и риски

## 1. Scope проекта

### 1.1 Auth

- получение auth token;
- проверка успешной авторизации;
- проверка неуспешной авторизации.

### 1.2 Read

- получение списка booking id;
- получение booking по id;
- проверка ответа для несуществующего id.

### 1.3 Create

- создание нового booking с валидным телом;
- проверка структуры ответа;
- проверка обязательных полей.

### 1.4 Update / Partial Update

- полное обновление booking;
- частичное обновление booking;
- проверка защищённых операций с токеном и без токена.

### 1.5 Delete

- удаление booking;
- проверка удаления;
- проверка защищённости удаления.

---

## 2. Что пока не входит

На первом этапе не входят:

- автоматизация API;
- нагрузочные проверки;
- security testing в глубину;
- SQL-проверки;
- локальный запуск платформы.

---

## 3. Основные риски

### 3.1 Риски авторизации

- токен может не выдаваться корректно;
- система может неверно обрабатывать невалидные креды;
- защищённые операции могут работать без авторизации.

### 3.2 Риски CRUD-логики

- booking может создаваться с некорректными данными;
- обновление может работать неполно или непредсказуемо;
- удаление может не удалять сущность фактически;
- система может возвращать неконсистентные данные.

### 3.3 Риски API-контракта

- неправильные статусы ответа;
- неполная или неожиданная структура ответа;
- отсутствие корректной обработки ошибочных запросов.
